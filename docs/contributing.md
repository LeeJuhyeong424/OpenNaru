# 기여자 가이드

OpenNaru에 기여해 주셔서 감사합니다.
이 문서는 외부 기여자와 내부 팀원 모두를 위한 가이드입니다.

## 기여 방법 요약

1. 이 저장소를 **Fork** 합니다.
2. `feat/기능명` 또는 `fix/버그명` 브랜치를 생성합니다.
3. 변경 사항을 작성하고 테스트를 통과시킵니다.
4. [Conventional Commits](https://www.conventionalcommits.org/) 형식에 맞게 커밋합니다.
5. **Pull Request** 를 제출합니다.
6. 리뷰 피드백을 반영하고 병합을 기다립니다.

## 이슈 작성 가이드

### 버그 리포트

- 버그가 발생하는 환경 (OS, 브라우저, 버전)
- 재현 단계 (번호 순서로 정리)
- 기대했던 동작 vs 실제 동작
- 오류 메시지 또는 스크린샷

### 기능 요청

- 해결하려는 문제 또는 필요성 설명
- 제안하는 해결 방법
- 대안이 있다면 함께 기술

## 커밋 메시지 컨벤션

```
<type>(<scope>): <한국어 요약>

[선택] 본문 — 변경 이유·상세 내용

[선택] footer — Closes #123
```

| type | 설명 |
|------|------|
| `feat` | 새 기능 |
| `fix` | 버그 수정 |
| `docs` | 문서만 변경 |
| `refactor` | 기능 변경 없는 코드 정리 |
| `test` | 테스트 추가·수정 |
| `chore` | 빌드 설정, 의존성 업데이트 등 |
| `perf` | 성능 개선 |

## PR 제출 규칙

- PR 제목은 커밋 메시지와 동일한 형식으로 작성합니다.
- PR 본문에 변경 사항 요약과 관련 이슈 번호(`Closes #이슈번호`)를 포함합니다.
- UI 변경이 있을 경우 스크린샷을 첨부합니다.
- 최소 **1명의 리뷰어 승인** 후 병합합니다.
- `pytest` 전체 테스트 통과 필수
- `ruff check .` 린트 오류 없음 필수
- 병합 방식: **Squash and Merge** (히스토리를 깔끔하게 유지)
- 병합 후 소스 브랜치는 자동 삭제됩니다.

## 코드 스타일

### Python (백엔드)

- 포매터·린터: **Ruff** (`ruff.toml` 설정 기준)
- 타입 힌트 사용 (가능한 경우)
- 함수·변수명: 영어 snake_case
- 주석: 한국어

### TypeScript (프론트엔드)

- 포매터: **Prettier**
- 린터: **ESLint** (Next.js 기본 설정)
- 컴포넌트명: PascalCase
- 함수·변수명: camelCase

## 행동 강령

모든 참여자는 상호 존중하는 태도로 소통해 주세요.
건설적인 피드백을 환영하며, 인신공격·비하 발언은 허용하지 않습니다.
